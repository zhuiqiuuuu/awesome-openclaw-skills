# 把 Claude Code 用成工程工具：8 条黄金法则与一套可复用工作流

> 来自 Eyad Khrais (CTO, 7年大厂经验) 的实战总结

## 🎯 前言

Claude Code 个人用起来很爽，但团队落地时会遇到：
- 换人就不稳定
- 换项目就复读  
- 产出质量时好时坏
- 返工成本很高

**核心问题**：没有把 Claude Code 当成"工程系统"来用，而是当成"高级搜索引擎"。

## 📋 太长不看版

1. **Plan Mode 是默认入口** - 先想清楚再动手
2. **CLAUDE.md 是项目章程** - 写"失忆后给自己的笔记"
3. **上下文 20-40% 开始退化** - 及时重置比硬撑更有效
4. **需求要写成接口** - 验收标准比形容词重要
5. **卡住就换变量** - 三次解释不通就改变策略
6. **聊天要升级成系统** - 做成可重复执行的流水线
7. **用模型梯队策略** - Opus规划，Sonnet执行
8. **建立持续优化飞轮** - 让系统自己进化

## ⚙️ 黄金法则详解

### 法则1：Plan Mode 是默认入口

> "你省下的不是5分钟，是一下午"

**问题**：直接给需求，AI可能选错技术路径，然后你开始追着修。

**解决方案**：
```text
连按两次 Shift+Tab 进入 Plan Mode
或使用标准模板：
```

**标准Plan模板**：
```markdown
目标（1句话）：
现状（它能看到什么/看不到什么）：
约束（必须遵守）：
- 技术栈/版本：
- 不能改的接口/行为：
- 性能/安全/合规：
验收标准（可验证）：
- 通过哪些测试/命令：
- 输出长什么样（示例/截图/路径）：
风险点（我最担心的1-3个点）：
请先给出方案对比与最终计划（含步骤与回滚），确认后再改代码。
```

**价值**：把自由度关进笼子里，避免歧义导致的返工。

### 法则2：CLAUDE.md 是项目章程

> "别把它写成说明书，写成'团队章程'"

**常见错误**：
- 写成README风格的项目介绍
- 科普基础知识
- 过于冗长失去重点

**正确写法**：
```markdown
# 项目目标
- 我们优先保证：正确性 > 可维护性 > 性能

# 代码与风格
- 只改与任务相关的文件，避免大范围重排
- 新增代码必须有对应测试（或解释为什么没有）

# 关键命令（必须用这些验证）
- 单测：pnpm test
- 类型：pnpm typecheck
- 格式化：pnpm format

# 已知坑（为什么要这么做）
- 不要直接改X：因为线上依赖了Y的边界行为
- 任何对外API的变更必须同时更新Z：否则会破坏移动端兼容
```

**4个原则**：
- ✅ **短**：必须记住的1页纸
- ✅ **项目特有**：写你们的现实，别科普
- ✅ **写清原因**：有"为什么"，执行质量高一个量级
- ✅ **持续更新**：同一个问题纠正两次就写进去

**小技巧**：工作时按`#`可快速追加规则到CLAUDE.md

### 法则3：上下文管理 - 20-40%法则

> "你不是输给模型，你是输给'污染的对话'"

**反直觉发现**：质量下降往往发生在你觉得"还很宽裕"的时候（20-40%），不是快满的时候。

**典型症状**：
- 忘掉早先确认过的约束
- 把旧方案当成新方案继续写
- 输出开始"像对但不对"，细节全是坑

**三大策略**：

#### 策略A：一事一聊（Scope）
```text
一个对话只做一件事：
- 一个feature
- 一次重构  
- 一次排错
不要在同一个对话里既做认证，又顺手重构DB，再顺便改CI
```

#### 策略B：外部记忆（External memory）
```text
让它把计划、关键决策、已验证结论写进文件：
- SCRATCHPAD.md
- PLAN.md
- DECISIONS.md
第二天回来直接读文件继续，别指望它"记得昨天聊过什么"
```

#### 策略C：复制粘贴重置法（Reset）
```bash
当对话开始变乱：
1. 复制关键信息
2. /compact 产生摘要
3. /clear 清空对话
4. 粘贴摘要和关键约束，重新开始
```

> **注意**：compact产生的摘要是给你复制出来用的'干净起点'，不是留在旧上下文里

### 法则4：需求接口化

**对比两种需求描述**：

❌ **模糊需求**：
```
"帮我做个登录系统"
```

✅ **接口化需求**：
```
"基于现有User模型实现邮箱+密码登录"
- Session存Redis，过期24小时
- 新增中间件保护/api/protected下所有路由
- 补齐单测：成功登录、过期、未登录访问被拒
```

**价值**：
- 砍掉自由度，减少返工
- 提前写验收标准
- 让AI理解边界条件

### 法则5：卡住就换变量

> "同一个点解释三次还不通，继续解释也不会更好——必须改变某个变量"

**四个"换变量"手段**：

1. **清空对话**：用`/clear`重新开始
2. **简化任务**：拆成最小可验证的一步
3. **给最小示例**：写正确输出让它照着扩展
4. **换表述角度**：比如改成"用状态机实现"而不是"处理这些分支"

### 法则6：聊天系统化

**三层进化路径**：

#### 第一层：可复用命令（Slash commands）
```bash
# 创建命令文件
.claude/commands/review.md

# 内容：标准化的代码审查检查项
```

#### 第二层：自动验证（Hooks）
```json
{
  "hooks": {
    "PostToolUse": [
      "pnpm test",
      "pnpm typecheck", 
      "pnpm format"
    ]
  }
}
```

#### 第三层：连接外部系统（MCP）
```json
{
  "mcp_servers": {
    "github": {
      "command": "npx",
      "args": ["@modelcontextprotocol/server-github"]
    }
  }
}
```

#### Headless模式
```bash
# 让Claude Code成为可脚本调度的组件
claude -p "执行代码审查并输出结果"
```

### 法则7：模型梯队策略

**推荐配置**：
- **Opus**：做规划、架构决策
- **Sonnet**：执行具体实现
- **CLAUDE.md**：保证两者在同一套约束下工作

### 法则8：持续优化飞轮

**自动化迭代**：
```
模型犯错 → 看日志 → 改CLAUDE.md/工具链 → 下一次更稳
```

**激进做法**：让Claude自己迭代优化CLAUDE.md，经过数月打磨系统表现提升一个量级。

## 📋 落地清单（10条）

1. ✅ 每次任务先用Plan Mode，产出"计划+风险+回滚"
2. ✅ 任何需求都写验收标准，最好能用命令验证
3. ✅ 把关键命令写进CLAUDE.md，规定"必须跑完再交付"
4. ✅ CLAUDE.md只写项目特有规则，每条带"为什么"
5. ✅ 一事一聊，超过一个主题就开新对话
6. ✅ 建SCRATCHPAD.md：记录计划、决策、已验证结论
7. ✅ 对话变乱就reset：`/compact`产摘要，`/clear`清空，再粘贴
8. ✅ 三次解释不通就停，换变量：清空、简化、示例、换角度
9. ✅ 把高频提示做成slash commands，减少临场发挥
10. ✅ 能自动跑的就自动跑：格式化、类型检查、单测放进hooks

## 🎯 最终目标

> **把Claude Code从"会写代码的搜索引擎"变成"需要约束、需要验证、需要沉淀规则的工程系统"**

当你建立这套体系后，Claude Code不再是"高级自动补全"，而是你的**生产力飞轮**：

- ✅ 产出更稳定
- ✅ 返工更少  
- ✅ 团队协作更顺畅
- ✅ 知识持续沉淀
- ✅ 系统自我进化

**记住**：模型能力越强，你越像**软件系统的责任人**，而不是**纯代码执行者**。
