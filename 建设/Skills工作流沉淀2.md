## 太长不看版

1.  **先想清楚再开口**：Plan Mode 不是可选项，是默认入口。
2.  **把规则写进 `CLAUDE.md`**：它不是文档，是"项目入职材料"，决定了每次对话的下限。
3.  **上下文在 30% 左右就开始退化，不是 100%**：作者的经验区间是 20%–40%；要学会控范围、用外部记忆、必要时果断清空重来。
4.  **把需求写成接口**：越具体越好，约束、验收标准、示例比形容词更重要。
5.  **卡住别硬怼**：三次解释还不通，就换变量：清空、简化、给最小示例、换表述角度。
6.  **把聊天升级成系统**：用 headless、hooks、slash commands、MCP 把高频动作自动化，形成飞轮。

## 1）Plan Mode：你省下的不是 5 分钟，是一下午

Plan Mode 的价值很朴素：**先把自由度关进笼子里**。

在 Claude Code 里连按两次 Shift+Tab 即可进入 Plan Mode。重点是先把"先规划、再动手"变成肌肉记忆。

### 一个能直接复用的 Plan 模板

```
目标（1 句话）：
现状（它能看到什么/看不到什么）：
约束（必须遵守）：
- 技术栈/版本：
- 不能改的接口/行为：
- 性能/安全/合规：

验收标准（可验证）：
- 通过哪些测试/命令：
- 输出长什么样（示例/截图/路径）：

风险点（我最担心的 1-3 个点）：

请先给出方案对比与最终计划（含步骤与回滚），确认后再改代码。
```

## 2）`CLAUDE.md`：别把它写成说明书，写成"团队章程"

### 写好 `CLAUDE.md` 的 4 个原则

- **短**：太长很容易开始选择性忽略。把它当成"必须记住的 1 页纸"。
- **只写项目特有**：别科普，写你们的现实。
- **写清原因**：同样一句规则，有"为什么"，执行质量会高一个量级。
- **持续更新**：同一个问题你纠正两次，就把它写进去。

### 一个最小可用的 `CLAUDE.md` 样例

```markdown
# 项目目标
- 我们优先保证：正确性 > 可维护性 > 性能

# 代码与风格
- 只改与任务相关的文件，避免大范围重排
- 新增代码必须有对应测试（或解释为什么没有）

# 关键命令（必须用这些验证）
- 单测：pnpm test
- 类型：pnpm typecheck
- 格式化：pnpm format

# 已知坑（为什么要这么做）
- 不要直接改 X：因为线上依赖了 Y 的边界行为
- 任何对外 API 的变更必须同时更新 Z：否则会破坏移动端兼容
```

## 3）上下文管理：你不是输给模型，你是输给"污染的对话"

**质量下降往往发生在你刚觉得"还很宽裕"的时候，而不是"快满了"的时候。**经验区间大约在 20%–40%。

### 三个立刻能落地的策略

**策略 A：一事一聊（Scope）**
一个对话只做一件事：一个 feature、一次重构、一次排错。

**策略 B：外部记忆（External memory）**
让它把计划、关键决策、已验证结论写进文件，比如 `SCRATCHPAD.md` / `PLAN.md`。

**策略 C：复制粘贴重置法（Reset）**
当对话开始变乱：
1. 复制你认为必须保留的关键信息
2. 用 `/compact` 让它压成短摘要
3. 用 `/clear` 清空对话
4. 把摘要和关键约束贴回去，再开始下一轮

## 4）把需求写成接口：越像 PRD，越像代码

### 对比一下这两种输入

**模糊需求**
- "做一个认证系统"

**可执行需求**
- "基于现有 `User` 模型实现邮箱+密码登录"
- "Session 存 Redis，过期 24 小时"
- "新增中间件保护 `/api/protected` 下所有路由"
- "补齐单测：成功登录、过期、未登录访问被拒"

## 5）当它卡住：别再加解释了，换一种打法

**同一个点解释三次还不通，就别解释了，直接换变量。**

### 四个"换变量"的手段

1. **清空对话**：用 `/clear` 重新开始
2. **简化任务**：把大任务拆成最小可验证的一步
3. **给最小示例**：写一个最小正确输出，让它照着扩展
4. **换表述角度**：比如改成"用状态机实现"，而不是"处理这些分支条件"

## 6）从聊天到系统：把高频动作自动化，才会越用越省心

### 第一层：可复用命令（Slash commands）
把你反复用的提示词写成命令文件，变成标准操作。

### 第二层：自动验证（Hooks）
它每次改完代码，你就自动跑：格式化、类型检查、单测。

### 第三层：连接外部系统（MCP）
把复制 GitHub issue、复制日志、复制数据库 schema 等操作自动化。

## 可以直接照抄的落地清单（10 条）

1. 每次任务先用 Plan Mode，让它先产出"计划 + 风险 + 回滚"。
2. 任何需求都写验收标准，最好能用命令验证。
3. 把你们的关键命令写进 `CLAUDE.md`，并规定"必须跑完再交付"。
4. `CLAUDE.md` 只写项目特有规则，每条尽量带"为什么"。
5. 一事一聊，超过一个主题就开新对话。
6. 建一个 `SCRATCHPAD.md`：记录计划、决策、已验证结论、下一步。
7. 对话开始变乱就 reset：`/compact` 产摘要，`/clear` 清空，再粘回摘要。
8. 三次解释不通就停，换变量：清空、简化、示例、换角度。
9. 把高频提示做成 slash commands，减少临场发挥。
10. 能自动跑的就自动跑：格式化、类型检查、单测放进 hooks。

## 最后一句话

如果你把 Claude Code 当成"会写代码的搜索引擎"，你会一直在修补输出。

**如果你把它当成一个需要约束、需要验证、需要沉淀规则的工程系统，它会变成你的生产力飞轮。**
