# 可复制的 Claude Code 配置方案

> 把 Claude Code 从个人技巧升级成团队资产

## 🎯 核心问题

很多团队用 Claude Code 会遇到：
- ✅ 个人用起来很爽
- ❌ 换人就不稳定
- ❌ 换项目就复读  
- ❌ 产出质量时好时坏
- ❌ 返工成本很高

**根本原因**：没有把经验沉淀成**可版本化、可评审、可迭代**的配置体系。

## 🏗️ 配置即代码理念

### 什么是"配置即代码"
不是简单放几个prompt文件，而是建立**AI编程的工程化接口**：

| 维度 | 传统做法 | 工程化做法 |
|------|---------|------------|
| **版本控制** | 口头约定 | Git版本化管理 |
| **Code Review** | 无 | Pull Request评审 |
| **回滚能力** | 手动恢复 | Git历史回滚 |
| **团队协作** | 个人经验 | 标准化配置 |
| **持续演进** | 被动更新 | 主动迭代优化 |

### 7构件配置体系
```
project/
├── CLAUDE.md          # 项目级记忆与约定（必须记住的1页纸）
└── .claude/
    ├── rules/         # 强制性规则（底线，必须遵守）
    ├── agents/        # 专用子代理（需要换脑子的任务）
    ├── commands/      # 高频斜杠命令（复杂流程一键化）
    ├── skills/        # 可复用方法论（领域知识沉淀）
    ├── hooks/         # 自动化守卫（经验教训→自动化）
    └── settings.json  # 配置文件（工具链集成）
```

**分层原则**：
- **CLAUDE.md**：讲"我们是谁"（项目特色）
- **rules/**：讲"必须守什么"（底线约束）
- **commands/**：讲"高频怎么做"（流程标准化）
- **hooks/**：讲"必须发生什么"（自动化保障）

## 📋 各构件详细说明

### 1. CLAUDE.md - 项目级记忆
> "不是入职文档，是失忆后给自己的笔记"

**错误写法**（像README）：
```markdown
# 项目介绍
这是一个电商系统...
技术栈：SpringBoot + MyBatis...
```

**正确写法**（像备忘录）：
```markdown
# 项目目标
- 我们优先保证：正确性 > 可维护性 > 性能

# 代码与风格  
- 只改与任务相关的文件，避免大范围重排
- 新增代码必须有对应测试（或解释为什么没有）

# 关键命令（必须用这些验证）
- 单测：pnpm test
- 类型：pnpm typecheck
- 格式化：pnpm format

# 已知坑（为什么要这么做）
- 不要直接改X：因为线上依赖了Y的边界行为
- 任何对外API的变更必须同时更新Z：否则会破坏移动端兼容
```

**4个原则**：
- ✅ **短**：必须记住的1页纸（不是百科全书）
- ✅ **项目特有**：写你们的现实，别科普基础知识
- ✅ **写清原因**：有"为什么"，执行质量高一个量级
- ✅ **持续更新**：同一个问题纠正两次就写进去

### 2. rules/ - 强制性规则
> "底线规则，短而硬"

**目录结构**：
```
.claude/rules/
├── security.md        # 安全红线
├── testing.md         # 测试要求
├── coding-style.md    # 代码风格
└── git-workflow.md    # Git工作流
```

**安全规则示例**：
```markdown
# 安全红线（违反直接阻断）
- 禁止硬编码密码、密钥、token
- 禁止生成SQL拼接，必须参数化
- 禁止暴露内部错误详情给前端
- 任何对外API变更必须review
```

**测试规则示例**：
```markdown
# 测试要求
- 新增代码必须有单测（除非明确说明为什么没有）
- 关键路径必须有集成测试覆盖
- 测试失败禁止合并
- 测试覆盖率不得低于70%
```

### 3. commands/ - 高频斜杠命令
> "把复杂流程变成一键操作"

**创建命令**：
```bash
# 创建代码审查命令
.claude/commands/code-review.md
```

**命令内容**：
```markdown
# /code-review - 标准化代码审查
请按以下维度审查代码：

## 功能正确性
- [ ] 实现了需求描述的所有功能点
- [ ] 边界条件处理完整
- [ ] 错误处理机制健全

## 代码质量  
- [ ] 符合项目编码规范
- [ ] 没有重复代码（DRY原则）
- [ ] 命名清晰，注释充分
- [ ] 圈复杂度不超过10

## 测试覆盖
- [ ] 新增代码有对应单测
- [ ] 测试用例覆盖正常和异常路径
- [ ] 所有测试通过

## 性能影响
- [ ] 没有明显的性能问题
- [ ] 数据库查询优化
- [ ] 避免N+1查询问题

请输出详细的审查结果和改进建议。
```

**推荐命令集**：
- `/plan` - 需求分析和方案设计
- `/code-review` - 标准化代码审查
- `/test` - 测试用例生成和验证
- `/refactor` - 代码重构建议
- `/security-check` - 安全漏洞扫描

### 4. agents/ - 专用子代理
> "需要换脑子的任务，交给专业agent"

**专用agent类型**：
```yaml
agents:
  planner:          # 方案规划与权衡
  code-reviewer:    # 代码审查专家
  security-reviewer: # 安全审计专家
  build-resolver:   # 构建排障专家
  e2e-designer:     # E2E测试设计专家
```

**代码审查agent示例**：
```markdown
# code-reviewer.md
你是资深代码审查专家，专注于：

## 审查维度
1. 功能正确性（是否完整实现需求）
2. 代码质量（规范、可读性、可维护性）  
3. 安全漏洞（注入、越权、泄露等）
4. 性能问题（算法复杂度、资源使用）
5. 测试覆盖（单测、集成测试）

## 输出格式
- 问题清单（按严重程度排序）
- 改进建议（具体可执行）
- 风险评估（业务影响分析）
```

### 5. hooks/ - 自动化守卫
> "把经验教训变成确定性行为"

**hook类型**：
```yaml
hooks:
  reminder:    # 提醒型（不阻断，提高体验）
  blocking:    # 阻断型（硬拦截，强制执行）
  consistent:  # 一致性型（自动格式化、检查）
```

**提醒型示例**：
```json
{
  "PreToolUse": {
    "npm-install": {
      "command": "echo '提醒：npm install 可能耗时较长，建议在tmux中运行'",
      "type": "reminder"
    }
  }
}
```

**阻断型示例**：
```json
{
  "PreToolUse": {
    "dangerous-rm": {
      "command": "if [[ $COMMAND == *'rm -rf'* ]]; then echo '禁止在生产环境执行rm -rf'; exit 1; fi",
      "type": "blocking"
    }
  }
}
```

**一致性型示例**：
```json
{
  "PostToolUse": {
    "typescript": {
      "command": "pnpm typecheck && pnpm format",
      "type": "consistent"
    }
  }
}
```

### 6. skills/ - 可复用方法论
> "领域知识沉淀，避免重复发明轮子"

**技能分类**：
```
.claude/skills/
├── tdd/                    # 测试驱动开发
├── backend-patterns/       # 后端设计模式
├── frontend-patterns/      # 前端设计模式
├── writing-guides/         # 技术写作规范
└── debugging-methods/      # 调试方法论
```

**TDD技能示例**：
```markdown
# tdd/red-green-refactor.md
## 红-绿-重构循环
1. **红**：先写失败的测试
2. **绿**：快速让测试通过（可以hack）
3. **重构**：消除重复，优化设计

## 原则
- 测试要小步快跑
- 每次只解决一个失败
- 重构时保持测试通过
- 频繁提交中间状态
```

## ⚠️ 常见坑与规避

### 1. CLAUDE.md写太长
**问题**：规则淹没规则，等于没有规则  
**解决**：狠删，把"必须发生"的动作迁到hooks

### 2. 把偏好写成底线
**问题**：团队有人不买账，落地失败  
**解决**：rules只写底线，偏好放CLAUDE.md或命令模板

### 3. Hook过度阻断
**问题**：一上来拦各种命令，体验崩了  
**解决**：先提醒型→再一致性型→最后阻断型

### 4. 密钥进仓库
**问题**：最危险也最常见的事故  
**解决**：用环境变量占位符，仓库只放示例

### 5. 照搬别人的流程
**问题**：项目结构不同，不适用  
**解决**：先抄结构，再抄模式，最后才抄实现
